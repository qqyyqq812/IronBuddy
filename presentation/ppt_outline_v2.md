## Slide 6 · 无线通信拓扑：UDP 透传 + IPC 心跳

**核心论点**：彻底抛弃 BLE 带宽瓶颈，走 WiFi UDP + 板内共享内存双层心跳。

**正文（4 条）**：
- **对外**：ESP32 → WiFi UDP → RK3399ProX（ASCII 16-bit 封包，~1 kHz 采样透传）
- **对内**：5 个 Python 进程通过 `/dev/shm/*.json` + atomic rename 交换（20+ 信号文件）
- **双网卡架构**：AP 热点接肌电（隔离 QoS 抖动）+ 外网接云端 LLM
- **绕 GIL 锁**：视觉进程内嵌 MJPEG:8080 服务器，彻底跳过 Flask GIL 竞争

**心路锚点（口播）**：
> "最开始以为 BLE 够用，结果 4 通道 1kHz 就撑爆；后来想 Python 多线程，发现 GIL 直接把视频流卡死，只能走 /dev/shm 文件握手。"

## Slide 7 · 技术路线·抉择①：视觉 — 本地 NPU vs 云端 GPU ⭐

**核心论点**：不是二选一，而是运行时热切换，各取所长。

**正文（对比表）**：

| 维度 | 本地 NPU（YOLOv5-Pose RKNN uint8）| 云端 GPU（RTMPose-m on RTX 5090）|
|---|---|---|
| 模型规格 | 640×640，uint8 量化 | FP32 浮点，高精度 |
| 推理时延 | ~107 ms/帧（板端） | ~30 ms + 网络 RTT |
| 置信阈值 | **0.08**（量化噪声限制） | **0.5**（业内标准）|
| 网络依赖 | 无 | 强（SSH 隧道）|
| 典型用途 | 断网兜底 / 零延迟 HDMI | 现场演示 / 高精度 |

**心路锚点（口播）**：
> "起初执着于"端上跑完一切"，结果量化模型置信度最高 0.2，用 0.5 阈值根本看不到人。又想全部上云，网一卡马上崩。最后做成写一个 `/dev/shm/vision_mode.json` 就热切换，甚至云端掉线自动降级到 NPU。"

**决策锚点**：`decisions.md §I` 双视觉模式 · 坑 4 keypoint sigmoid · 坑 9 量化角度跳变

**配图方案（截图 + 生成）**：
- **S04**：板端 HDMI 直连的骨架实时识别截图（蹲至谷底瞬间抓帧）
- **G06**：双引擎架构图 —— 左侧"本地 NPU 路径"（摄像头 → RKNN → 骨架）+ 右侧"云端 GPU 路径"（摄像头 → HTTP POST → RTMPose → JSON 回传）+ 中间"信号文件热切换"节点

## Slide 8 · 技术路线·抉择②：算法 — 端到端 vs 中继融合 ⭐

**核心论点**：多模态硬塞板端必崩溃，Mid-Fusion 先降维后融合才救活。

**正文（架构对比）**：

**❌ 端到端双分支（原型期，已废）**：
- 骨架点张量 + 原始 EMG 波形 → 双路注意力网络 → 分类
- 结果：模态结构天然不对齐 · 4 GB 内存几秒 OOM · 无法收敛

**✅ Mid-Fusion 中继融合（定型）**：
- 视觉侧：骨架 → 关节角度 + 相位索引（抛弃像素）
- 肌电侧：原始波 → Biquad 带通 → FFT 频域 → 滑动 RMS（纯标量）
- 融合：7D 特征（角度/角速/角加速/目标 RMS/代偿 RMS/对称分/相位）→ GRU(hidden=16) → 3 头输出
- **模型体积：1488 参数 · < 1 MB · CPU 微秒级推理**

**心路锚点（口播）**：
> "我一开始真的就是"万物皆可深度学习"地把所有张量塞进去，板子 3 秒 OOM。痛定思痛改用领域知识先降维，才跑通。这是整个项目最大的教训：边缘算力不是无底洞。"

**决策锚点**：`decisions.md §VIII` FSM + GRU 双引擎 · 报告第 3.2 节

**配图方案（生成）**：
- **G07**：Mid-Fusion 流水线图
- 左起：摄像头（图像）→ 关键点 → 角度/相位；ESP32（原始 EMG）→ 带通 → FFT → RMS
- 中：7D 特征向量框
- 右：GRU(16) → 三头（相似度/分类/相位）

## Slide 10 · 技术路线·抉择④：LLM — 单轨集中 vs 双轨双核 ⭐

**核心论点**：一个 LLM 做不了"实时陪练+长期教练"两件事，拆成两条线。

**正文（分工对照表）**：

| 维度 | DeepSeek 前端实时 | OpenClaw 后端常驻 |
|---|---|---|
| 角色 | 语音陪练 / 按钮问答 | 日报 / 周报 / 偏好画像 |
| 触发 | "教练" 语音 / 疲劳 1500 / UI 按钮 | cron：09/20/23 点 + shm trigger |

## Slide 12 · 落地实证②：MVC 动态校准 + 域对齐

**核心论点**：人体阻抗差 3×，绝不能硬编码阈值 —— 每个人都要校一遍"我的 100%"。

**正文（两阶段方案）**：
- **阶段①：动态 MVC 校准（个体归一化）**
  - 用户点按钮 → 3.5 秒最大发力 → 取峰值 RMS 作为 `_MVC_VALUES`
  - 公式：`rms_raw_pct = raw_rms / MVC_peak × 100`，钳位 [50, 2000]，失败回落硬编码 400
- **阶段②：硬件域对齐（模型域适应）**
  - ESP32 廉价硬件 vs MIA Delsys 医疗级 → 线性滤镜 `MIA = α·User + β`
  - 本地实测：target α=2.12 β=-21.4 / comp α=2.64 β=-6.56（基于 5700 点 ↔ 6000 点 p05/p95 对齐）
  - 不动 MIA 权重（val acc 94.4%），在输入侧"骗过"模型

**心路锚点（口播）**：
> "临床上 MVC 是金标准，但我们一开始觉得麻烦就硬编码 400 —— 结果瘦子全是 0，健身老哥全饱和。加上 3 秒峰值法才解决，顺便还做了硬件域对齐。"

**决策锚点**：`decisions.md §XIV 决策 12` · 现状快照 V4.6 硬件域对齐

**配图方案（截图 + 生成）**：

## Slide 13 · 落地实证③：语音交互 + 无触 UI

**核心论点**：健身环境不允许触屏 —— 放弃 Vosk、抛弃 edge-tts，落到百度 AipSpeech。

**正文（方案选型表）**：

| 候选 | 放弃原因 |
|---|---|
| Vosk（本地离线）| Debian 10 ARM64 `libvosk.so` glibc ABI 不兼容，缺 `_ZNSt7__cxx11...` 符号 |
| edge-tts（微软云）| 板端网络抖动时完全不可用 |
| **百度 AipSpeech** ✅ | 稳定 TTS + STT，延迟 < 500 ms，免费额度够用 |

**关键特性**：
- 自适应 VAD（baseline RMS + 动态阈值 + 1.2 s 静音停顿）
- ALSA 错误日志屏蔽（ctypes 劫持 `libasound.so`）
- 语音命令库：静音 / 切换动作 / 调疲劳上限 / 飞书推送
- **无触交互**：CSS 玻璃拟态只读 UI，配置全部走语音 / 按钮

## Slide 14 · 技术栈闭环与知识盲区反思

**核心论点**：从 C 语言 DSP 到 Web PWA，全栈贯通；也暴露了两个核心盲区。

**正文（左栈图 + 右反思）**：

**左：全栈闭环**
- 硬件层：差分 PCB + ESP32 C + 锂电管理
- 信号层：Biquad IIR + FFT + 滑动 RMS（DSP）
- AI 层：YOLOv5-Pose RKNN + GRU 7D 三头（PyTorch → NPU）
- 系统层：5 进程 + /dev/shm + systemd-ready
- 应用层：Flask REST + SSE 流式 + PWA + 飞书 API

**右：两个盲区**
1. **算力规划缺位**：原型期端到端 OOM 是整个项目最大教训
2. **通信队列理论薄弱**：Python GIL 卡顿查了很久才定位到 `/dev/shm` 非阻塞心跳
