# 1. 项目背景从计次到智能交互

## 1.1 现状痛点与目标受众
- **痛点**：传统居家健身应用仅能进行简单的 reps（个数）计算，无法感知肌肉的真实疲劳状态，且反馈滞后。
- **初衷**：为居家健身人群提供一个“免穿戴、零门槛”的实时矫正环境。

## 1.2 核心价值：生理感知导向
- **范式转移**：通过增加肌电传感系统，系统能够实时反馈肌肉激活度，将计算逻辑从“做了多少口”升级为“练得怎么样”。
- **智能闭环**：结合 DeepSeek API 语音交互，教练能根据动作相似度给出实时纠偏建议。
- **训练闭环**：生成的个性化计划能通过飞书 (Feishu) 即时推送，形成闭环管理。

# 2. 环境突破：零路由器的 ICS 物理直连

## 2.1 针对常规连接方式的局限性创新
老师提供的参考资料主要依靠路由器中转，但在实验室或户外等无路由环境下，开发板极易失联。
- **创新点**：自主验证并实现基于 **Windows 网络共享 (ICS)** 的网线直连开发模式。
- **优势**：无需额外网络硬件，高带宽稳定传输，物理直达。

---

## 2.2 物理链路与 IP 捕获 (具体操作)
1. **物理连接**：开发板 -> 网线 -> 电脑网口。
2. **软件配置**：在 `ncpa.cpl` 中，将当前已连外网的网关（如 WiFi）“共享”给对应有线网卡。
3. **精准寻址**：
   ```powershell
   # 在 PC 端 PowerShell 执行，捕获 192.168.137.x 网段动态 IP
   arp -a
   ```
   排除网关 `.1` 后，即可定位开发板最新的 IPv4 地址，实现快速 SSH 破门。

---

## 2.3 硬件初始化的“避坑”逻辑：25s 延时保护
为解决 USB 总线枚举晚于网络请求导致的开机失联，我们在 `/etc/rc.local` 中对老师的方法进行了时序加固：
- **初始化命令**：
  ```bash
  # 关键动作：预留物理枚举时间
  sleep 25
  # 强制拉起网卡并执行 DHCP 请求
  ip link set wlan0 up
  dhclient wlan0
  ```
- **核心逻辑**：这 25 秒的等待是系统稳定性的基石，避免了由于底层硬件就绪滞后而产生的 DHCP 挂死问题。

**方案 B：Python Web 推流（现役方案）**

## 3.2 蜂鸣器：GPIO 控制

<!--

采用 `sysfs` 接口操作 GPIO 153 引脚控制蜂鸣器：
```bash
echo 153 > /sys/class/gpio/export
echo out > /sys/class/gpio/gpio153/direction
echo 0   > /sys/class/gpio/gpio153/value   # 拉低 → 蜂鸣
echo 1   > /sys/class/gpio/gpio153/value   # 拉高 → 静音
```
开机时由 `buzzer_init.sh` 将引脚初始化为高电平（静音状态）。状态机检测到违规半蹲时，由 Python 调用 Shell 脚本短暂拉低电平触发警报。

> 配置参考：`RK3399proX/3399 gpio 参考及例程/Rockchip_Developer_Guide_IO_CN.pdf`

## 3.3 语音系统：麦克风输入与音箱输出

系统复用板载声卡，通过 ALSA 路由 `amixer sset 'Playback Path' SPK` 将输出切换至外接小音箱。

**语音输入**：`voice_daemon.py` 持续录音监听，内置唤醒词模糊匹配（"教练"/"叫练"等），识别后进入语音转文字流程，将用户语音转化为文本写入 `/dev/shm/chat_input.txt`。

**语音输出**：`tts_daemon.sh` 轮询 `/dev/shm/llm_reply.txt` 和 `/dev/shm/chat_reply.txt`，有新内容时调用 `edge-tts`（声线：`zh-CN-YunxiNeural`）合成 MP3，再通过 `mpg123 -a plughw:0,0 -r 16000` 播放。

> 关键细节：硬件 I2S 锁定 16kHz 采样率，直接播放 48kHz 的 TTS 音频会导致声卡死锁或严重变调。`-r 16000` 参数强制执行软件重采样，解决了这一问题。

# 4. 神经网络选型：YOLOv5-Pose

## 4.1 为什么选择 YOLOv5 而非 v7 / v8？

| 考量维度 | YOLOv5 | YOLOv7/v8 |
|---------|--------|-----------|
| RKNN 转码成熟度 | 社区已大量验证，`.rknn` 模型直接可用 | 转码支持尚不完善，需自行适配 |
| 预训练姿态模型 | 开源库中有成熟的 17 点人体骨骼模型 | 需额外训练或转换 |
| 端侧帧率 | 在 RK3399ProX 上稳定 15-20 FPS | 算力需求更高，帧率不达标 |

## 4.2 模型部署方式
使用开源预训练好的 YOLOv5-Pose 进行 17 点人体骨骼关键点检测。模型已转换为 `.rknn` 格式，由 C++ 推理引擎 (`main`) 加载运行，通过共享内存将检测结果（关键点坐标 JSON + 骨骼叠加图片）输出供 Python 层消费。

# 5. 主控状态机：深蹲检测逻辑

## 5.1 V2 趋势检测状态机
不使用绝对角度阈值判定"站直/下蹲"，而是检测角度的**变化趋势**。取过去 8 帧数据构成滑窗，计算斜率方向：

```
状态流转：NO_PERSON → IDLE → DESCENDING → ASCENDING → IDLE
```

- 角度波动 < 20° 持续 25 帧 → 判定为 `IDLE`（稳定站立）
- 斜率显著负向 → `DESCENDING`（正在蹲下）
- 斜率由负转正 → 转折点，进入 `ASCENDING`（正在起身）

这种基于趋势的设计避免了单帧数值震荡造成的误判。

---

# 5. 主控状态机（续）

## 5.2 深蹲质量判定
在趋势从下降转为上升的转折瞬间，读取本次动作中的**最低膝盖角度**：
- 最低角 < 90° → 记为 **标准深蹲** ✅
- 最低角 ≥ 90° → 记为 **违规半蹲** ❌，同时触发蜂鸣器警报

计数间设有 1.5 秒冷却期，防止单次动作被重复计数。

## 5.3 触发方式
系统**不会自动触发**大模型点评。只有在以下情况下才会向云端发送训练数据：
- 用户通过语音说出唤醒词"教练"进行对话
- 用户在前端网页点击"生成本组点评"按钮

这确保了用户对训练节奏的完全控制。

# 6. 云端大模型对接与语音交互

## 6.1 OpenClaw 网关与 DeepSeek API
板端通过 SSH 反向隧道将本地 18789 端口映射至 WSL 宿主机上的 OpenClaw 网关。`openclaw_bridge.py` 实现 WebSocket 长连接，通过 Token 鉴权后建立通信通道。

每次对话请求使用独立 UUID 标识，采用异步 Future 匹配响应，避免阻塞主循环。发送给大模型的 prompt 不包含任何原始坐标数据，仅包含精简的统计信息（标准次数与违规次数）。

## 6.2 语音对话流程
1. `voice_daemon` 检测到唤醒词 → 录音 → ASR 转文字 → 写入 `/dev/shm/chat_input.txt`
2. `main_claw_loop.py` 轮询该文件 → 组装 prompt → 通过 OpenClaw 发送
3. 收到回复 → 写入 `/dev/shm/chat_reply.txt`
4. `tts_daemon.sh` 检测到新回复 → edge-tts 合成 → 音箱播放

# 7. 系统总架构与全局链路

系统按物理位置分为三层：

| 层级 | 位置 | 运行内容 |
|------|------|---------|
| **边缘感知层** | RK3399ProX 板端 | C++ NPU 推理引擎、Python 推流中台、FSM 状态机、语音守护、蜂鸣器/音箱控制 |
| **指挥控制层** | 宿主机 (Windows/WSL) | OpenClaw 网关、SSH 反向隧道、代码开发与同步、浏览器前端展示 |
| **云端智能层** | DeepSeek API | 接收精简训练战报、生成教练点评与训练计划、飞书 WebHook 推送 |

板端与云端之间**不直接通信**。所有大模型请求经由宿主机的 OpenClaw 网关中转，板端通过 SSH 隧道访问网关的本地端口。

# 8. V2 升级：肌肉激活热力图

## 8.1 从"动作对不对"到"肌肉练没练"

V1 只能判定深蹲是否到位。V2 引入 **视觉-生物力学混合管线**，告诉用户哪块肌肉在发力、发力多少，并检测肌肉代偿。

```
摄像头 → NPU 2D检测(25fps) → 2D→3D Lifting(CPU) → 关节角度 → 肌肉激活估算 → SVG热力图
```

| 组件 | 选型 | 理由 |
|------|------|------|
| 2D 姿态 | YOLOv5-Pose (NPU) | V1 已验证，零改动 |
| 2D→3D | VideoPose3D (ONNX, CPU) | 轻量 16.9M 参数，causal 实时 |
| 生物力学 | 力矩臂查表 + 静态优化 | 无需训练，基于解剖学文献 |
| 前端 | SVG 人体热力图 | 浏览器原生，轻量 |

---

## 8.2 生物力学引擎（V2.1 累积模式）

### 技术架构

| 模块 | 功能 | 关键参数 |
|------|------|----------|
| `lifting_3d.py` | VideoPose3D ONNX 推理 | 243帧窗口，板端 ~465ms/次 |
| `joint_calculator.py` | 8 关节 3D 角度 + 角速度 | 膝/髋/肘/肩 × 左右 |
| `muscle_model.py` | 13 肌群累积激活估算 | 标准 rep +8%，违规 +3% |
| `exercise_profiles.json` | 动作→肌群映射 | 深蹲、哑铃弯举 |

### 累积模式工作原理
- 每完成一次标准动作 → 主动肌 **+8%**，协同肌 +5%，稳定肌 +3%
- 每完成一次违规动作 → 主动肌 +3%（毕竟做了动作）
- 约 **13 次标准深蹲**主动肌达 100%（满载红色）
- 代偿检测：稳定肌激活 > 主动肌 50% 时触发警告

# V2.2 迭代升级

## 三项核心架构改进

| 改进 | 旧方案 | 新方案 | 收益 |
|------|--------|--------|------|
| **离线 ASR** | Google Speech API（需外网） | Vosk 本地推理 (~50MB 模型) | 唤醒 3-8s → **<1s**，完全离线 |
| **3D 角度判定** | 2D 投影角（视角敏感） | VideoPose3D 3D 真实角 | 准确率 ~75% → **~95%** |
| **视频传输** | JS 链式 fetch 轮询 | MJPEG multipart 流 | 消除 GC 抖动，浏览器原生解码 |

## 智能教练 Prompt 升级

**V2.1**：仅发送「标准 N 次 + 违规 M 次」→ 泛泛鼓励
**V2.2**：注入合格率 + TOP3 肌肉激活 + 最近 3 天训练历史 → **个性化点评**

```
本组数据：标准 15 次（合格率 83%），肌肉 TOP3：quadriceps(92%)、glutes(88%)、hamstrings(45%)
最近3天：03-22 标准20违规5 → 03-23 标准25违规3 → 今日 标准15违规3
→ "今天节奏不错但量少了点，违规率继续下降说明动作越来越标准，下组试着多做5个。"
```
