5. **节奏**：每章讲清楚"做了什么 → 遇到什么坑 → 怎么解决的"

## 第2章：网络配置与远程通信（大改）

### 当前问题
- 只有3步表格太简略
- 没讲清楚"这是我们在老师方法基础上的创新"
- 缺少具体命令和避坑细节

### 修改方案（扩充为4页）

**第1页 - 背景与创新点**：
- 老师的PDF教程提供了3种连接方式：串口、路由器转接、手机热点
- 但没有"网线直连电脑"这种最简单的方式
- 我们验证了Windows ICS (Internet Connection Sharing)直连方案
- 好处：不需要路由器，一根网线就够

**第2页 - ICS直连操作步骤**：
```
1. 网线连接：板子网口 ↔ 电脑网口（或USB网卡）
2. Windows设置：ncpa.cpl → WiFi适配器 → 属性 → 共享 → 勾选"允许其他网络用户通过此计算机的Internet连接来连接" → 选择有线网卡
3. 板子自动获取 192.168.137.x 段IP
4. PC端查找板子IP：
   arp -a
   在 192.168.137.x 段找到非 .1 的地址就是板子
5. SSH连接：
   ssh toybrick@192.168.137.xxx
```

**第3页 - 25秒延时保护**：
- 问题：板子开机后USB总线枚举比网络请求慢，导致wlan0还没准备好就尝试连WiFi，结果连接失败
- 解决：在 `/etc/rc.local` 中加入延时
```bash
sleep 25           # 等USB/WiFi模块完成初始化
ip link set wlan0 up
dhclient wlan0     # 此时再请求DHCP，百分百成功
```
- 这25秒是我们反复测试出来的经验值，少了会失败

**第4页 - IP变化与日常使用**：
- 提醒：每次换网络环境IP都会变
- 常用命令：
  - 板端查IP：`ip addr show wlan0 | grep inet`
  - PC端扫描：`arp -a` 或 `nmap -sn 192.168.137.0/24`
- 连通后可以用VS Code Remote-SSH + Agent进行开发（附截图）

**3.2 蜂鸣器 → 小音箱（重写）**：

改标题为 "3.2 声音反馈：从蜂鸣器到小音箱"

内容：
- 最初方案是用GPIO控制无源蜂鸣器做警报
- GPIO 153引脚编号是从Rockchip的GPIO参考文档查到的
- sysfs接口操作（保留原有代码块）
- 测试结果：蜂鸣器确实响了，但声音太单调，只能"嘟——"一声
- 最终方案：换成外接小音箱 + edge-tts语音合成
- 这样可以播报具体内容，比如"这个深蹲没蹲到位，注意膝盖角度"

```bash
# 蜂鸣器测试（已弃用，但成功驱动过）
echo 0 > /sys/class/gpio/gpio153/value   # 响
echo 1 > /sys/class/gpio/gpio153/value   # 停

# 现役方案：小音箱播放
amixer sset 'Playback Path' SPK          # 切换到外接音箱
mpg123 -a plughw:0,0 -r 16000 reply.mp3  # 播放TTS音频
```

**3.3 语音系统（补充踩坑）**：
- 保持现有唤醒词+ASR+TTS流程
- 重点补充：
  - I2S声卡锁定16kHz采样率的坑
  - edge-tts默认输出48kHz MP3，直接播放会导致：要么声卡死锁、要么声音变调像"花栗鼠"
  - 解决方案：`mpg123 -r 16000` 强制软件重采样
  - 唤醒词为什么设了13种变体：因为ASR在嘈杂环境下识别率不高，"教练"可能被识别成"叫练"、"交练"等，我们全部纳入匹配

## 第4章：神经网络选型（中改）

### 当前问题
- 只讲了YOLOv5选型，没提V3已经改用云端RTMPose

### 修改方案

**4.1 保持原有YOLOv5 vs v7/v8对比表**

**4.2 NPU的现实（新增一段）**：
- 原计划：YOLOv5-Pose模型转RKNN格式，在板端NPU上跑
- 实际结果：RKNN量化后模型精度严重下降，置信度普遍<0.3，基本不可用
- 原因：INT8量化对姿态估计的关键点精度影响很大

**4.3 V3方案：云端RTMPose（新增）**：
- 改用RTMPose-m模型，部署在AutoDL云端的RTX 5090上
- ONNX Runtime GPU推理，通过HTTPS直连（不走SSH隧道）
- 板端用USB摄像头拍照 → POST到云端 → 返回17关键点JSON
- 往返延迟约100ms，完全满足实时需求

| 方案 | 精度 | 延迟 | 可用性 |
|------|------|------|--------|
| 板端RKNN NPU | 差 (conf<0.3) | ~50ms | 不可用 |
| 云端RTMPose HTTPS | 好 (conf>0.8) | ~100ms | 稳定可用 |

## 第9章：拓展模块（新增）

这是最后一章，给大家介绍我们后续增加的新能力。

### 9.1 真实肌电传感器对接
- 之前的肌肉激活数据是模拟计算的，V3接入了真实的ESP32双通道肌电传感器
- 硬件：ESP32 + AD8232模块，通过WiFi以UDP:8080发送数据
- 数据格式：两个传感器同步采样，每次发两个数值，空格分隔
- DSP处理管线：20Hz高通 → 50Hz陷波（去工频） → 150Hz低通 → RMS包络
- 传感器放置：肱二头肌（目标肌群）+ 前臂桡侧（代偿检测）
- 发现：使用甩力弯举时前臂代偿特别明显，这验证了我们的放置方案

### 9.2 GRU训练闭环
- 实际录制了哑铃弯举的三类动作数据：
  - golden（标准）：全程控制，肘关节固定
  - lazy（代偿）：借助甩力，前臂明显发力
  - bad（不到位）：角度不够，发力不足
- CompensationGRU模型：7维特征 → 30帧滑窗 → 输出相似度+三分类
- 工具链：采集(collect) → 验证(validate) → 训练(train) → 部署(deploy)
- Streamlit可视化面板：数据探索、训练曲线、混淆矩阵、实时推理

### 9.3 云端GPU协同
- AutoDL RTX 5090 部署RTMPose-m ONNX推理服务
- 板端通过HTTPS直连云端（无需SSH隧道）
- 一键部署脚本：rsync同步代码 + scp同步模型 + SSH启动5个服务
- 开发机到板端到云端的完整链路打通

### 9.4 管理面板
- 开发了Web管理面板（/admin），集成：
  - 系统总览：板子在线状态、训练数据统计、模型信息
  - 服务管理：一键启动/停止5个板端服务
  - 训练数据浏览：按动作类型和标签分类查看CSV文件
  - 系统状态：GPU使用情况、板端CPU温度、网络配置
  - 项目信息：Git版本、提交历史、模型参数
