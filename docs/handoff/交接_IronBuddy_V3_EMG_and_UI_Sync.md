# IronBuddy V3 UI状态同步与DSP肌电重构交接文档

> 生成时间：2026-04-04 15:30

## 【项目当前状态】
目前 IronBuddy V3.0 的后端视觉 FSM 推理管线与肌电 DSP 滤波管线**已经完全拉通并稳定运行**。前端 UI 渲染由于前期遗留的 Web Worker 跨域与强缓漏洞导致了界面的“冻结假死”，现已彻底查明根因并于 `index.html` 完成了全面修复。当前，只要被测者进入镜头，系统可以完美无缝更新计步数字并实时高频刷新 1000Hz 下采样后的肌电波形动画。

## 【已解决的核心问题】
在本次会话中，我们重点攻克和收尾了以下深层通信与处理屏障：
1. **纯肌电时间的频域切除手术 (DSP)**: 我们接管了 ESP32 裸发的高频 UDP 肌电波流，原生手工构建了 `20Hz 高通 + 50Hz 极点陷波 + 150Hz 低通` 的级联 Biquad IIR 滤波器（见 `udp_emg_server.py`），通过翻折叠加 100 毫秒滑动窗口完美抑制了因为接触不良导致的剧烈自激振荡与工频噪点。
2. **移除 UI 动态连杆的致命时序异常**: 修复了旧版肌肉条移除后 `KinematicsRig.updateSquatPose` 查询不到相应 DOM `[body, thigh, calf]` 节点导致的 JS 主线程灾难性崩溃问题，引入了非空包裹判断。
3. **Blob Worker 相对域解析黑洞**: 我们侦破了为什么 FSM 服务器（板载 `10.105.245.224:5000`）完全健康并持续输出状态 JSON，但前端页面永远只显示“等待中”与“连接断开”的诡异 BUG。这是因为由于主控轮询引擎是从 `Blob` 生成的 Web Worker，浏览器解析相对路径 `/state_feed` 时重定向到了无此路由的 `blob:` 域名导致瞬间抛出 NetworkError。我们在所有相关的 `fetch` 动作中补充强制拼接了 `location.origin`，并加上了 `{ cache: 'no-store' }` 避免激进的盘内缓存。

## 【原生指令与红线约束】
1. **原生指令**: "我现在用ESP32通过udp向我的电脑以每秒1000个数据的速度传输...请你写接口文件" / "把传输 IP 改板子...我们需要在右下角显示波形" / "不发力的时候波形幅值很高...怀疑是转模拟信号引起的肌贴松动...使用傅立叶变换定位过滤" / "现在就是记不了数...为什么喇叭有声音报警，但左上角一直显示等待转态...要求所有的音响、麦克风 CPI 全回归正常" 
2. **红线与约束**: 
   - **绝对禁止随意使用相对路径进行 `fetch`**: 为了兼容 Blob Worker，在任何异步前端 `index.html` 通信修改中强制打上 `location.origin`。
   - **禁止移除/阻塞 DSP 运算**: 由于 NPU 不擅长跑连续的高频 DSP 以及 Python 性能拉胯，必须保留现有架构中基于独立 `threading.Thread` 的无锁共享队列模式，并在 `main_claw_loop` 外层获取。
   - **防卡死底线**: 由于板载环境极其脆弱（NPU 跑推理占用 80% 负载），对于任何新功能切忌再跑重量级计算在 Flask 或 Socket 中，严格遵循解耦策略。
