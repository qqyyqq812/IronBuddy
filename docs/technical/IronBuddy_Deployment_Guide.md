# IronBuddy 纯软落板与前沿联调：防坑排雷图鉴

本指南由 Worker_3 归档沉淀，提纯了 V1 到 V3 版本推进过程中的核心踩坑记录与解决思想。在后续开发、调试或现场答辩时，如遇玄学 Bug，可优先翻阅此册。

## 📍 篇章一：通信架构防坑录 (UDP vs TCP)

> **背景**：ESP32 贴片采集高精尖肌电电流时，采样频率高达惊人的 `1000Hz`（即 1ms 发一个数值）。初版曾设想在 ESP32 上跑 WebSocket 或 TCP，惨遭失败。

*   **玄学 Bug：画面抽搐与越延迟越卡**
    *   **排雷机制**：TCP 在遭遇无线电磁干扰时，具有“丢包即阻塞死等”的特性（队头阻塞）。如果丢了 10ms 前的电压包，TCP 会在之后 100ms 里疯狂申请重传，致使这期间产生的所有新鲜血液（实时的姿态肌电）全面断供塞爆，反馈到网页上就是“卡顿一秒，随后加速播放两秒”。
*   **破局架构 (高频无状态弹射)**：
    *   强制改用纯 **Linux UDP 多播/点对点发包**。UDP 的不担保特性极其契合该场景——“丢了就丢了，我只要最新的一帧”。这极大降维打击了延迟问题。

## 📍 篇章二：前端并线与心跳防死锁机制

> **背景**：初期 FSM (状态机) 会用慢速（2fps，500ms）从 `/dev/shm` 抽取肌电数据并下发前端。

*   **玄学 Bug：视觉连坐效应 (一死全死)**
    *   **排雷机制**：如果摄像头丢失或者 NPU 死机，FSM 循环即刻停摆，不再往前端推送 json。导致的结果是，明明贴片还在正常采集高潮肌电，前端却没有丝毫更新。
*   **破局架构 (双轨制与极速前端接管)**：
    *   **解耦并路**：让 NPU 和 FSM 只走 `/state_feed` 的慢车道负责更新骨架 `angle`。
    *   让前端 `index.html` 亲自开设 30Hz（33ms）的高速轮询线程，直插 `/api/muscle_activation` 摄取肌电信号，即使挂起 FSM 仍能跑赢全局。
*   **看门狗熔断拦截 (150ms Sentinel)**：
    *   若肌电数据因底层进程死锁连续 150 毫秒没有变动（连底噪都一样），触发前端 DOM 本地渲染红色非遮挡 `⚠️ 传感器失联` 弹窗。**绝对不去封锁**其他 3D 和视频管道。

## 📍 篇章三：基于 NPU 的“三重防鬼影”哲学

> **背景**：RK3399Pro 内置 NPU 硬件抽取二维骨骼点常由于杂乱的室内背景（如扫把、晾衣架）抽出虚假的异形关节。

*   **破局架构 (The Triple Filter)**：
    1.  **置信度阈值斩断 (Confidence)**：忽略神经输出层 `< 60%` 把握的游离坐标。
    2.  **生理结构反曲判定 (Physiological Constraints)**：对传入的大腿-小腿向量夹角施加刚性算力锁死。深蹲夹角绝不低于 40°、不高于 190°，一旦越界认定为鬼影，强制复用上一帧 `fsm.angle`，防止小人乱跳。
    3.  **C 位画幅锁死 (BBox Prominence)**：多具检出人形时，只认“中心面积最大的 Bounding Box”，摒弃背景路人的杂乱干扰。

## 📍 篇章四：无机纯软“神隐验证阵”开发流

> **背景**：设备不在身边，在课堂等极其简陋的开发环境，如何验收前端页面动态响应和 3.0 SVG Rig 重构逻辑？

*   **破局架构 (Bash Mock Simulator)**：
    *   在服务器开一列后台死循环管线模拟 UDP 数据流（每 `0.01s` 向 `/dev/shm/muscle_activation.json` 打入 `echo RANODM`），并利用 Python 脚本循环注入符合余弦震荡 `125 + 55 * math.cos()` 函数的 `angle` 深蹲模拟角度数据。
    *   完美复刻板端真实微秒级文件 IO。前台所见即真机，零负担上板联调。
