# IronBuddy V3.x 硬件扩展：美诺韦(Menovomed) 表面肌电系统集成方案 (Implan)

## 🎯 目标描述
用户决定选用**美诺韦医疗（山西迈尔诺）**的无线表面肌电贴片作为 IronBuddy 的肌肉激活检测硬件。
本案旨在规划如何将这类**专业级/封闭生态的医疗传感器**数据，低延迟、稳定地接入到跑在 RK3399ProX 开发板上的 IronBuddy 核心生物力学管线中。

*(注：用户提供的链接 `id=549` 为 SENIAM 国际标准的肌电贴片贴放解剖学指南，验证了该产品走的是专业化运动康复路线。)*

## 🔌 一、 硬件与通信链路嗅探层 (Data Link Layer)

专业多通道肌电仪的采样率通常高达 1000Hz~2000Hz，由于数据吞吐量极大，美诺韦的传输方案通常是以下三种之一。我们在 RK3399ProX 上需要采取对应的接入策略：

### 方案 1：私有 2.4G USB 接收器（最高概率 🌟🌟🌟）
- **形态**：传感器发货时自带一个类似无线鼠标的 USB Dongle。
- **RK3399ProX 接入法**：将 Dongle 插在板子的 USB 3.0/2.0 口。Linux 内核通常会将其识别为虚拟串口（Virtual COM Port）。
- **设备路径**：通过 `ls /dev/ttyUSB*` 或 `ls /dev/ttyACM*` 访问。
- **波特率**：极高（如 115200 或更典型的 921600 / 2000000）。

### 方案 2：原生蓝牙 / BLE (Bluetooth Low Energy)
- **形态**：贴片直接作为标准的 BLE 外设广播。
- **RK3399ProX 接入法**：启用板载蓝牙（需确认 RK3399ProX 板载的天线已连好）。
- **工具栈**：使用 Python 的 `bleak` 库扫描其 MAC 地址，订阅特定的 GATT Characteristic (Notify 流)。

### 方案 3：底层基站/网关网络转发 (TCP/UDP)
- **形态**：传感器先连到一个“网关盒子”或其官方的 Windows 软件工作站。
- **RK3399ProX 接入法**：在同一局域网下，让官方软件开一个 TCP Server 或 UDP 广播，RK3399 用 Python Socket 写个 Client 实时抓取流。

## ⚙️ 二、 软件架构融合 (Integration Architecture)

一旦确立了物理链路，在 IronBuddy 系统侧将新增一个独立的采集守护进程，与现有的推流和 FSM 互不干扰。

### 1. 新增采集守护进程 `menovomed_daemon.py`
在 `hardware_engine/sensor/` 目录下新建该守护进程。
**核心职责**：
- **串流解码**：根据美诺韦官方协议（后续需向厂家索要），将连续的 Hex（十六进制）字节流按包头、包尾、校验和进行解包，提取出 CH1~CH8 的 ADC 原始电压值。
- **信号预处理**：由于原始 sEMG 信号正负交替且含噪音，守护进程先做**全波整流 (Rectification)** 和 **低通滤波/均方根 (RMS, 步长 50ms)** 的包络线提取，算出绝对肌电强度（mV）。
- **写入共享内存**：以 `20Hz` 的 Freq 将过滤后的肌电特征值写入 `/dev/shm/sensor_data.json`。

### 2. 生物力学引擎融合 `muscle_model.py`
在 `on_rep_completed()` 结算时：
- 从 `/dev/shm/sensor_data.json` 读取特定部位（如股四头肌贴片）的 RMS 微伏峰值。
- 设定 MVC（最大自愿收缩）校准百分比。肌电强度直接作为**真实肌肉发力得分 (Real Activation Score)**，取代过去纯靠关节角度推测的面条人渲染。

### 3. FSM 动作抖动监控 `main_claw_loop.py`
- 如果同时含有 IMU 数据，抓取 Gyro（陀螺仪）的高频震荡。当用户力竭时出现肌肉震颤，触发 DeepSeek 语音提醒。

## 🆚 核心释疑：肌电 (sEMG) 与 加速度 (IMU) 是否重复？

用户提问：“既然有了表面肌电，还有没有必要再加加速度传感器，这两个会不会重复？”
**结论：完全不重复，两者是完美的正交互补关系。**

| 传感器类型 | 美诺韦肌电 (sEMG) | 加速度/陀螺仪 (IMU) |
|------------|------------------|--------------------|
| **测什么** | **神经驱动与肌肉收缩 (生物电)** | **肢体空间姿态 (物理运动)** |
| **解决痛点** | 解决“假发力”。能精确知道你用了几成力，判定是否力竭。 | 解决“摄像头被挡住或视角偏差”。能输出绝对下蹲角度。 |
| **盲区** | 无法知道你的肢体在什么空间位置（例如：站着绷紧大腿和深蹲到底，肌电都可能很高）。 | 无法知道你是否在发力（例如：被人抬着腿被动运动，IMU有数据但肌电为零）。 |
| **融合意义** | 提供受力与疲劳基准。 | 提供标准的空间轨迹基准。 |

> **硬件排查建议**：现在许多高端医疗贴片（如 Delsys Trigno 或 傲意）在出厂时，**同一个贴片里已经同时内置了 EMG 电极和 6轴/9轴 IMU**。
> 您需要向美诺韦确认：**他们这款传感器的硬件手册里，是否自带了加速度计/陀螺仪输出？**
> - **如果自带**：完美！我们用一套协议同时接收电信号和角度信号，不用额外买任何东西。
> - **如果没带（纯肌电）**：建议在另一个部位（如腰部）低成本补一个几十块钱的 IMU，用以纠正摄像头的 3D 深度误差。

## 🧪 三、 实施路径与行动项 (Call to Action)

此为闭源商业产品的典型接入流程，我们需要**用户协助完成 Phase 1 才能写代码**：

**Phase 1：协议获取与联调阻抗 (用户行动项 🚩)**
由于美诺韦未在官网公开底层通信代码，**您需要联系他们的对口销售或技术支持（Menovomed），索取以下资料：**
1. **《二次开发接口文档》或《通讯协议文档》**（最重要：我们需要知道数据包的首尾标志、字节序、缩放系数）。
2. **连接方式确认**：请问厂家这套系统是“蓝牙直连”还是“带 USB 接收器”？
3. **是否存在 Linux / Python SDK？** (有则直接调库，无则我们手写 Socket/Serial 解析)。

**Phase 2：离线模拟与解析器编写 (Agent C 行动项)**
1. 获取协议后，Agent C 将立刻使用 Python 编写 `menovomed_parser.py`。
2. 造一份伪造的 Hex 字节流，验证解包、校验和(Checksum)与 RMS 滤波算法的正确性。

**Phase 3：板端集成 (联合行动项)**
1. 更新 `deploy/ironbuddy-supervisor.conf`，将 `menovomed_daemon` 设为开机自启和死锁自动重启。
2. 用实际设备绑在腿上做 5 个深蹲，调优 MVC 映射阈值。
