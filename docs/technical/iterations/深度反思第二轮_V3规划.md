## 一、现状审视：我们在行业坐标中的位置

### 竞品对标

| 维度 | Tonal | Tempo Mirror | Forme Life | **IronBuddy** |
|------|-------|-------------|------------|---------------|
| 硬件 | 专用壁挂+数字阻力 | 独立式柜体+杠铃 | 壁挂镜+双滑轮 | **通用 SoC+USB 摄像头** |
| 传感 | 17 个力传感器 | 3D ToF 传感器 | 双 RGB 摄像头 | **单 RGB + NPU 2D→3D** |
| 动作纠正 | 数字阻力自适应 | 骨架隐私化+实时文字 | 双摄角度分析 | **蜂鸣器+TTS 语音** |
| 动作种类 | ~340 种 | ~60 种 | ~100 种 | **2 种（squat+curl）** |
| 教练 | 预录视频+AI引导 | 预录视频+社区 | 真人远程教练 | **LLM 按需点评** |
| 游戏化 | 成就+连续天数 | 类排行榜 | 进度追踪 | **无** |
| 月费 | $49 | $39 | $39 | **免费（API 成本 ~¥0.1/次）** |

**结论：** IronBuddy 最大优势是**成本极低**（~¥300 硬件 vs 竞品 $1,500+）和**LLM 个性化教练**。最大短板是**动作种类**和**游戏化缺失**。

## 二、九项水面之下的改进机会

### 🔴 架构级问题

#### 1. FSM 硬编码严重（V3.0-A1）
**现状：** `SquatStateMachine` 中 `ANGLE_STANDARD=90`、`TREND_WINDOW=8`、`IDLE_FRAMES=25` 全部硬编码。新增动作（如俯卧撑）需复制整个类。

**应改为：** 动作配置驱动的通用 FSM。每个动作定义：
- 关注的关键点集合
- 各阶段角度阈值
- 状态流转规则

```python
# exercise_profiles.json 扩展
"pushup": {
  "fsm": {
    "tracked_angle": ["l_elbow", "r_elbow"],
    "standard_threshold": 90,
    "trend_window": 6,
    "idle_frames": 20
  },
  "muscles": { ... }
}
```

**影响：** 从 2 → **无限动作扩展**，是产品化关键。

---

#### 2. 通信模式碎片化（V3.0-A2）
**现状：** 前后端通过 **7 个独立 HTTP 轮询** 通信：
- `/state_feed`（FSM，1500ms）
- `/api/muscle_activation`（肌肉，2000ms）
- `/api/chat_reply`（对话回复，2000ms）
- `/api/chat_input`（语音输入，2000ms）
- `/api/chat_draft`（识别草稿，1000ms）
- `/llm_reply_feed`（LLM 总结，3000ms）
- `/video_feed`（MJPEG 流，V2.2 已改）

**应改为：** 单一 WebSocket 通道推全部实时数据。

```
ws://board:5000/ws → { type: "state", ... }
                   → { type: "muscle", ... }
                   → { type: "chat_reply", ... }
```

**收益：** 
- 网络连接数 7 → **1**
- 延迟从 1.5-3s 轮询间隔 → **实时推送 <100ms**
- 服务端资源压力大幅下降

### 🟡 功能层提升

#### 3. 肌肉模型线性累积过于简化（V3.0-B1）
**现状：** `on_rep_completed()` 中 primary +8、synergist +5、stabilizer +3，13 次达 100%。**完全线性**，不考虑：
- 动作质量分级（不只是"合格/违规"二分）
- 下蹲深度影响（蹲到 60° vs 89° 的肌肉激活差异很大）
- 节奏与离心控制（慢下快上 vs 快处理很不同）

**学术依据：** 2024 研究表明，基于 sEMG 的肌肉激活估算中，**离心-向心速度比**是区分训练效果的关键因子。IronBuddy 已有关节角速度数据（`velocities`），但当前仅在瞬时检测中使用，未反馈到累积模型。

**改进方向：** 
```python
# 从二分法 → 连续质量分
quality_score = f(bottom_angle, descent_speed, ascent_speed, symmetry)
# 0.0 (极差) → 1.0 (完美)
increment = base_weight * quality_score  # 非固定值
```

---

#### 4. 动作 Profile 仅两种（V3.0-B2）
**现状：** `exercise_profiles.json` 仅 28 行，只有 `squat` 和 `bicep_curl`。

**应至少支持 6 种常见徒手动作：**
- 俯卧撑（肘关节+肩关节）
- 仰卧起坐（髋关节+脊柱）
- 弓步蹲（膝关节不对称）
- 站姿提踵（踝关节）
- 开合跳（全身协调性追踪）
- 平板支撑（定时类而非计数类）

**关键挑战：** 俯卧撑需要侧面视角才能看到肘部弯曲。当前系统假设正面摄像头，需要在 FSM 中添加**视角自适应**逻辑。

---

#### 5. 教练缺乏渐进式知识构建（V3.0-B3）
**现状：** 每次对话都是无记忆的一次性 session（`generate_new_session=True`）。教练无法记住"上次说过要注意膝盖内扣"、"你已经连续进步 3 天了"。

**应改为：** **跨 session 记忆注入**。不需要改 DeepSeek 会话管理，而是在 prompt 中注入 condensed 历史：

```python
# 从 training_log.json 提取关键事件
memory_lines = extract_recent_events()
# "3月22日: 首次做到20个标准蹲\n3月23日: 违规率首次降到10%以下"
prompt = f"{soul_text}\n\n你对这个学员的记忆:\n{memory_lines}\n\n当前训练数据:..."
```

### 🟢 体验层优化

#### 6. 零游戏化 = 零粘性（V3.0-C1）
**行业数据：** 游戏化健身应用的用户留存率比非游戏化高 **48%**。IronBuddy 当前完全没有：
- ❌ 连续训练天数（streak）
- ❌ 成就系统（首次 20 个标准蹲、合格率首次 100%）
- ❌ 进度里程碑（勋章/等级）

**最小可行游戏化：** 
```json
// achievements.json
[
  {"id": "first_perfect", "name": "零违规", "desc": "单组 100% 合格率", "icon": "🏆"},
  {"id": "streak_7", "name": "周末战士", "desc": "连续训练 7 天", "icon": "🔥"},
  {"id": "total_100", "name": "百蹲勇士", "desc": "累计 100 次标准深蹲", "icon": "💪"},
  {"id": "muscle_max", "name": "满载红灯", "desc": "主动肌达到 100%", "icon": "🔴"}
]
```

---

#### 7. 动作实时纠正缺失（V3.0-C2）
**现状：** 蜂鸣器只在"违规半蹲完成后"才响——已经做完了才告诉你不对。

**理想交互：** 在**下降过程中**检测到可能的违规趋势时，提前用 TTS 纠正：
- "再低一点" → 下降速率放缓且角度还在 100° 以上
- "膝盖不要超过脚尖" → 膝盖 x 坐标 > 脚尖 x 坐标
- "背挺直" → 肩-髋连线偏离垂直 > 30°

**技术可行：** FSM 的 `DESCENDING` 状态中已有实时角度和趋势数据，逻辑增量极小。

---

#### 8. SVG 热力图粗糙（V3.0-C3）
**现状：** 前端肌肉热力图使用矩形色块拼接，视觉上像"色块表格"而非"人体"。

**提升方向：** 使用 SVG `<path>` 绘制解剖学人体轮廓。开源参考：
- human-body-outline SVG（CC0 许可）
- 分区域着色（渐变填充 + stroke 描边）

---

#### 9. NPU 量化 VideoPose3D（V3.0-D1）
**现状：** VideoPose3D 跑在 CPU 上，~465ms/次。RK3399Pro NPU 的 2.4 TOPs INT8 算力完全闲置于此。

**量化路径：**
```
videopose3d_243f_causal.onnx
  → rknn-toolkit 混合量化（全连接层 INT8，BatchNorm 保持 FP16）
  → .rknn 文件部署
  → 预计延迟降至 30-50ms
```

**挑战：** VideoPose3D 含 temporal convolution，RKNN 对 Conv1D 支持有限，可能需要 reshape 为 Conv2D + squeeze。
