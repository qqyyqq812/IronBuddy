## §1 抢话式长对话

### 背景

当前 V4.5 是**单轮问答**模式（喂"教练" → 录一次 → 回复 → 退出）。抢话式（barge-in）支持：
- 连续多轮对话，不必重复唤醒词
- TTS 正在播放时，用户可直接说话打断当前播报

### 设计

1. **后台持续录音**
   - `arecord` 作为**后台守护**持续运行（不在 `record_with_vad` 内反复启停）
   - 避免 V4.5 坑 10 的僵尸进程独占 `/dev/snd` 问题
   - 通过环形缓冲（ring buffer）保存最近 5 s 音频

2. **播放时并发收音**
   - `aplay` 播放 TTS 时，`arecord` 同时收音
   - 方案 A：独立声卡（USB 麦 + 板载扬声器各走一路）
   - 方案 B：ALSA `dsnoop` / `dmix` 共享单卡

3. **能量检测打断**
   - 后台线程检测用户声音能量 > 阈值（e.g. baseline + 800）
   - 触发 `killall -9 aplay` → 清空剩余 TTS 队列
   - 状态机立即进入"监听中"

4. **分段 TTS（Streaming TTS）**
   - DeepSeek `_sync_chat` 改 **SSE 流式输出**每 200 ms chunk
   - `voice_daemon.py` 接收每 chunk → 立即合成 + 播放（分段 TTS）
   - 首字延迟从当前 ~3 s 降至 ~500 ms

### 工作量

- 预估：**2 - 3 天**
- 影响文件：
  - `hardware_engine/voice_daemon.py`（~100 行改动）
  - `hardware_engine/cognitive/deepseek_direct.py`（SSE stream 消费）

### 风险

- `arecord + aplay` 并发可能触发声卡冲突（见 `docs/technical/decisions.md` 坑 10 的教训）
- ALSA 共享 PCM 在 RK3399ProX 上**未验证**，可能需要降级到双卡方案
- VAD 阈值在 TTS 播放时需动态调整（避免机器自己的声音触发打断）

## §2 DeepSeek 语音流式 SSE

### 背景

- 当前：前端 `triggerDeepseek` POST 后，每 2 s 轮询 `/api/deepseek_reply`
- 问题：端到端延迟 2 - 4 s，用户体验卡顿

### 目标

SSE 从后端推 reply chunk，前端实时显示打字机效果。

### 设计

- `streamer_app.py` 新增 `/api/deepseek_stream` SSE endpoint
- 后端转发 DeepSeek `stream=True` 返回的 chunk
- 前端 `EventSource` 监听，每 chunk 追加到对话面板

### 工作量

- 预估：**半天**
- 影响文件：`streamer_app.py` + `templates/index.html`

## §3 飞书双向对话

### 背景

- 当前：仅**板端 → 飞书**单向推送（用 `tenant_access_token` 发到群）
- 目标：**飞书群内发消息"教练 XXX"** → 板端接收 → 生成回复

### 设计

   - 订阅群消息 webhook（需要飞书开放平台配置 webhook URL）

2. **板端接收端**
   - `streamer_app.py` 新增 `/api/feishu/webhook` 端点
   - 接收飞书消息 → 写 `/dev/shm/chat_input.txt`
   - FSM 的 `_chat_handler` 已经消费 `chat_input.txt`，**链路打通**

### 工作量

- 预估：**1 天**（主要是飞书后台配置 + 内网穿透）

### 风险

- 生产环境需要 **HTTPS + 签名校验**（飞书强制要求）
- 内网穿透（frp / ngrok）在实验室可用，演示环境需稳定公网 IP

## §5 日志 rotation

### 背景

- 当前：`/tmp/*.log` **无限增长**
- 风险：24/7 连续运行 7 天，日志可能爆盘（`/tmp` tmpfs 容量有限）

### 目标

每 100 MB 或每天切分日志。

### 设计

- **方案 A（Python 原生）**：`logging.handlers.RotatingFileHandler(maxBytes=100*1024*1024, backupCount=3)`
- **方案 B（systemd）**：用 journald 替代 `/tmp/*.log`，自带 rotation

### 工作量

- 预估：**半天**
- 影响文件：`hardware_engine/*.py` 的 logging 初始化段

## §6 长线程心跳检测 + 自恢复

### 背景

5 进程（vision / streamer / fsm / emg / voice）任一崩溃，需要人工 SSH 重启。

### 目标

每 30 s 检查 5 进程存活，**崩溃自动 `nohup` 重启**。

### 设计

- 新增 `scripts/healthcheck.sh`
- `cron` 每分钟跑一次，逻辑：
  ```bash
  pgrep -f "[m]ain_claw_loop.py" > /dev/null || {
    echo "[$(date)] main_claw_loop.py crashed, restarting..." >> /tmp/healthcheck.log
    nohup python hardware_engine/main_claw_loop.py > /tmp/fsm.log 2>&1 &
  }
  # ... 其他 4 进程同理
  ```

### 工作量

- 预估：**1 小时**
- 影响文件：`scripts/healthcheck.sh`（新增）+ crontab

### 风险

- 崩溃循环（crash loop）：如果进程启动就崩溃，cron 会一直重启。需要加**指数退避**或**3 次失败后告警**

## §7 动态 MVC 校准（实时重算硬件域 α/β）

### 背景

V4.6 已通过 [`tools/hardware_domain_calibrate.py`](../../tools/hardware_domain_calibrate.py) 做了一次**离线校准**（2026-04-12 三次实测 CSV × MIA 200 clip），产物：
- target α=2.12 β=-21.4
- comp α=2.64 β=-6.56

但这组系数是**静态**的——每次换人 / 换贴片位置 / 传感器电池电压变化，硬件响应都会漂移。

### 设计：60 秒自校准流程

1. **前 10 s 静息**：传感器贴好、身体放松，脚本记录 RMS 底噪 → 新 `user_p05`
2. **中 20 s 缓慢渐进发力到 MVC**：循序渐进到最大自愿收缩，记录 → 新 `user_p95`
3. **后 30 s 休息 + 自动重算**：脚本调 `hardware_domain_calibrate.py --user-stats-override '{"p05":X,"p95":Y}'` 写新 JSON
4. **`udp_emg_server` 热加载**：inotify 监听 `domain_calibration.json` 或每 30 s mtime 检查，重装载 `_DOMAIN_CALIB`

### 需改

- **新增** `tools/live_mvc_calibrate.py`：现场流程向导 + 实时 RMS 聚合 + 写 JSON
- **新增** 前端按钮 "重新校准硬件域"（触发板端跑向导）
- **改** `udp_emg_server.py` 加 inotify 监听或 mtime 轮询

### 估时

1-1.5 天（核心是前端交互 + 流程稳定性，不是算法）。
