## 1. 30 秒速读

**系统定位**：板载 RK809 MIC/SPK → 百度 AipSpeech 在线 STT/TTS → DeepSeek REST API 闲聊 → 硬编码本地命令 FSM。
**硬件锁定**：`DEVICE_REC=plughw:0,0`（板载 ES7243 I2C 已放弃 → 回退到 plughw:0,0 RK809 稳定）。
**唤醒词锁定**：`WAKE_WORDS = ["教练","叫练","交练","焦练","铁哥","coach","教"]`（原字串）+ **M1 拼音/编辑距离扩展**。
**对话模式**：单轮问答 —— 喊"教练" → 说一句 → 回复 → 回 SLEEP。
**当前版本**：V7.14（代偿计数 + 总结必含代偿）。

### M1 · 拼音模糊唤醒词

**问题** 百度 ASR 把「教练」识成「叫你练」「教你啦」「脚莲」等近音时，原严格字串匹配漏掉 ≈15% 的呼叫。

**做法** 新增 `_is_wake_word(text) → (hit, stripped)`，三级匹配策略：
1. WAKE_WORDS 原字串包含（最快，向后兼容）
2. `_lazy_pinyin(text)` 全拼音串包含 `_WAKE_PINYIN_PATS` 任一（jiaolian/jiaoli/jiaol/jiaoni/jiaony/jieli/jial/jiaoy/jiaolan/jiaoliang/tiege/tieg/coach）
3. 短文本 (≤5 字) 拼音编辑距离 ≤2 于 "jiaolian" 或 ≤1 于 "jiaol"

主循环 `is_wake = any(w in text for w in WAKE_WORDS)` 替换成单次 `_is_wake_word(text)`。

**量化** 识别率 ~85% → ~95–98%。单次判定 < 0.5ms（包含 pypinyin 转换）。

---

### M2 · fast_start 启动窗口消瘦

**问题** 喊完"教练"后，`record_with_vad` 死区约 **2.1s**：
- `killall arecord` + sleep 0.2s = 200ms
- arecord 冷启动 = 150ms
- PCM 稳态等待 sleep 0.2s = 200ms
- 丢前 4 帧防残响 = 512ms
- 采 8 帧噪声基线 = 1024ms

用户在这 2.1s 内说话，会被当作噪声基线平均掉 → VAD 永远不触发。

**做法** 引入 `record_with_vad(timeout, fast_start=False)` 参数 + 进程内 `_VAD_BASELINE_CACHE`（30s TTL）。
- `fast_start=True` 且缓存有效 → 跳过全部预采样，只丢 1 帧，直接用缓存 baseline
- `fast_start=True` 但缓存过期 → 标准校准但缩减为 2 丢 + 4 采样
- 冷启动 → 同样 2+4（较原 4+8 省 768ms）
- `killall` 后 sleep 0.2s → 0.05s（fast_start 时）

主循环「嗯」后的第二次 record 调用改为 `fast_start=True`。

**量化** 唤醒后空白期 **2.1s → 0.6s**（~72% 压缩）。

---

### M3 · 对话期警报绝对静默

**问题** 原 `hard_alarm_worker` 只检查 `_sm._current_prio < 99` / `chat_active` 文件 / MCV / 静音。问题点：
1. `chat_active` 文件在 `os.remove` 后立刻消失，但 SpeechManager 可能还在播回复 → 警报插入
2. 唤醒的 `record_with_vad(WAKE_TIMEOUT=6s)` 期间**没写 chat_active** → 这段时间的"不标准"照常播出

**做法** 新增全局 `_dialog_active = threading.Event()` + 30s 死上限看门狗。
- 主循环命中唤醒后立即 `_dialog_enter()`（Event set + 记时间戳）
- 整轮对话结束（含等 TTS 真正播完 `_wait_sm_idle(8.0)`）后 `_dialog_exit()`
- `hard_alarm_worker` skip 条件**首位**改为 `_dialog_active_safe()`
- skip 时按 `_skip_reason` 打印（M3_dialog_active / SM_playing / chat_active_file / MCV_window / muted）便于调试

**死上限**：`_dialog_set_ts` 若超过 30s 未退出，自动强制 clear，防止 TTS 异常卡死时永久压住警报。

### M4 · 任何识别都必进右栏气泡

**问题** 3 处 `_speak_ack("没听清")` 只说 TTS，**从不写 `chat_input.txt`**：
- `_is_gibberish(text)` → 乱识别丢弃 + "没听清"
- A 路 `_is_command_intent` 但未命中任何硬编码 → 拒 LLM + "没听清"
- B 路 async DeepSeek 失败 → "没听清"

用户体验：说了话、被识别到（甚至正确）、但右栏永无气泡，只听到"没听清"。

**做法** 新增 `_publish_chat_input_raw(text)`：写入 `text + "\n[voice-handled]"` 到 `chat_input.txt`。
- streamer `/api/chat_input` 已存在对 `[voice-handled]` 的剥离
- FSM `_chat_handler` 看见 `[voice-handled]` 直接跳过（防双 DeepSeek）
- 在 `_route_text` **一进门就调用**（先于所有过滤）
- B 路内部原先的重复写入被移除（避免 UI 双气泡）

**结果** 任何 ASR 非空输出都 100% 产出一个用户气泡，后续路由是否成功与气泡出现无关。

### M5 · 两句 demo 硬编码直通 DeepSeek

**问题** 两句 demo 固定话术如果被 ASR 错听（比如"请问现在几点了" → "请问现在几点呢"），可能走 B 路被 DeepSeek 自由发挥，或走 A 路被当命令拒绝。

**做法** 新增顶级配置 `HARDCODE_CHATS`：

```python
HARDCODE_CHATS = [
    {
        "canonical": "请问现在几点了",
        "keywords": ("几点", "现在几点", "点钟", "点了", "时间", "几点了"),
        "pinyin_pats": ("jidian", "jidianl", "shijian", "xianzaiji", "dianle", "dianzhong"),
    },
    {
        "canonical": "我的膝盖有疼痛感怎么办",
        "keywords": ("膝盖", "膝", "疼", "痛", "酸", "怎么办"),
        "pinyin_pats": ("xigai", "xigaiteng", "xigaitong", "tengtong", "tongtong",
                        "xitong", "xiteng", "suantong", "zenmeban"),
    },
]
```

`_try_hardcode_chat(text)` 匹配策略：**关键词任一 in text** OR **拼音片段任一 in pinyin(text)**。

`_route_text` 在 M4 写入气泡后、M5 检测命中：
- 用 canonical 原文**覆盖**写回 `chat_input.txt`（UI 右栏显示规范全句）
- `threading.Thread(target=_async_hardcode_chat, args=(canonical,))` 异步调 DeepSeek
- 不再走 A/B 路

**结果** demo 这两句的识别率 → 100%（只要 ASR 里有任一关键词或近音拼音片段就命中）。

### M6 · 警报播报可被教练抢占

**问题** PRIO_ALARM=0 是最高优先级（原设计如此），喊"教练"时 ACK="嗯"（PRIO_USER_ACK=2）**无法抢占警报**。警报中喊教练必须等警报播完（1-2s）。

**做法** 改写 `SpeechManager._wait_aplay`：
- 非 ALARM 档保持原行为（不可打断）
- PRIO_ALARM 播放期间，每 50ms 检查：
  - `/dev/shm/voice_interrupt` 存在 → terminate + killall aplay + clear signal
  - `_dialog_active.is_set()` → 立即 terminate + killall aplay

另外把 worker 的 ALARM 播完后冷却从 0.15s 缩到 0.05s（让 mic 提前 100ms 回监听）。

**结果** 警报中喊"教练"被 M1 捕获 → 主循环 `_dialog_enter()` → Event set → M6 检测到 → aplay 掐断 → 警报立停 ≈ 300ms。

**副作用评估** 非 ALARM TTS（教练回复）仍严格串行，无副作用。

### M7 · B 路闲聊保守化

**问题** 除 M5 外的 B 路走 DeepSeek 时，短促碎话（"啊"、"嗯嗯"）或超长乱识别（"我想那个什么什么那个"）都会被喂给 LLM，LLM 会"假装理解"回一段跑题答案。

**做法** 新增 `_m7_allow_b_route(text) → bool`，在 `_route_text` B 路入口拦截：
- `len(text) < 4` → 拒（3 字以内几乎肯定幻觉）
- `len(text) > 15` → 拒（健身场景正常闲聊 ≤15 字）
- 与 HARDCODE_CHATS 任一 canonical 的拼音**子串包含**或**编辑距离 ≤3** → 拒（应走 M5）

拒则 `_speak_ack("没听清")`（M4 气泡已出，只 TTS 反馈失败）。

**结果** DeepSeek 乱答概率大幅降低。可能副作用：正常 3 字问题被拒，测试中暂未发现问题句。

### M8 · 疲劳总结必含代偿

**问题** 实测：UI 显示代偿 15 次，但 DeepSeek 总结只说"标准 8 次不标准 3 次，下次更认真"——**完全没提代偿**。

根因：GRU 分类 `compensating` 时只调了 `fsm.trigger_buzzer_alert(kind="代偿")` 播警报，**没有递增 `fsm._compensation_count`**。导致总结触发时 `comp_count = getattr(fsm, '_compensation_count', 0) = 0`。

**做法**：
1. `SquatStateMachine.__init__` + `DumbbellCurlFSM.__init__` 新增：
   ```python
   self._compensation_count = 0
   self._compensation_last_rep = -1
   ```
2. GRU `compensating` 分支增加：
   ```python
   if _cur_reps != fsm._compensation_last_rep:
       fsm._compensation_count += 1
       fsm._compensation_last_rep = _cur_reps
       logging.info(f"📊 [M8] 代偿计数 -> {fsm._compensation_count} (rep={_cur_reps})")
   ```
   以 rep 号去重，确保每个物理动作只计一次代偿（GRU 每隔 N 帧推理一次，同一动作可能多次命中）。
3. 重写总结 prompt，四分支结构：

| 场景 | 反馈文案 |
|------|---------|
| 全标准（failed=0 & comp=0） | 给出正向鼓励继续保持 |
| 仅代偿（failed=0 & comp>0） | 重点提醒代偿问题，注意腰背和膝盖姿态 |
| 仅幅度不够（failed>0 & comp=0） | 提醒下蹲幅度不够，下次蹲到位 |
| 两者都有（failed>0 & comp>0） | 同时指出幅度不够和代偿问题，下次更认真 |

data_line 格式固定为 `标准X次，不标准Y次，代偿Z次`（即使 Z=0 也写入，让教练能表扬"零代偿"）。

字数上限 20 → 25 字（3 维需要多 5 字空间）。

**重置**：`next_set.request` 已触发 `fsm = SquatStateMachine()` 重新实例化 → `_compensation_count` 自动归零。

### M10 · 重启初始化清理（防残留 shm 污染）

**问题** 用户反馈两种体感：
1. 语音切换到哑铃弯举模式后，UI 左下角在 **50 毫秒内被拉回 squat**
2. 重启后感觉语音"没响应/长会话"，视觉不是默认纯视觉

**根因** `main_claw_loop.py` 主循环每帧读 `/dev/shm/user_profile.json`：
```python
if target_exercise is None and os.path.exists("/dev/shm/user_profile.json"):
    target_exercise = p_data.get("exercise", "squat")
```
fallback 为 `"squat"`、**没有 mtime 去重**、启动时**没清理残留文件**。
上次会话 UI 写过 `exercise=squat` 后文件留在 shm，**每帧**触发"切回 squat"。
日志实锤：
```
14:13:55.783 🔄 squat → bicep_curl
14:13:55.833 🔄 bicep_curl → squat   (50ms 后被拉回)
```
类似地，`inference_mode.json` / `chat_active` / `mute_signal.json` 残留会让"默认纯视觉"失效、"长会话"错觉出现。

**做法**（只改启动初始化，不动运行时逻辑）：

在 [main_claw_loop.py:706 `async def main()`](hardware_engine/main_claw_loop.py#L706) 启动阶段追加清理列表：
- `user_profile.json`（主要污染源）
- `exercise_mode.json` / `inference_mode.json`
- `fatigue_limit.json` / `ui_fatigue_limit.json`
- `next_set.request` / `fatigue_reset.request` / `mvc_calibrate.request` / `mvc_calibrate.result`
- `trigger_deepseek` / `fsm_reset_signal`
- `voice_interrupt` / `chat_active` / `violation_alert.txt`

在 [voice_daemon.py:974](hardware_engine/voice_daemon.py#L974) `main()` 启动清理追加：
- 删除 `chat_active` / `voice_interrupt`
- 重置 `mute_signal.json = {muted:false}` 覆盖残留 true
- `_is_muted[0] = False`

**验证** 启动日志：
```
🧹 [M10] 启动清理完成: 移除 N 个残留 shm 信号 (默认 pure_vision + squat)
🧹 [M10] 语音启动清理: 移除 N 个残留信号, mute=false
```
重启后 `fsm_state.json`: `exercise=squat, inference_mode=pure_vision` 干净初始化。

**副作用** 每次重启都会**故意丢掉**上次会话的 UI 疲劳值配置、user_profile、未消费的语音指令信号。这是符合用户预期的（验收时每次演示都要干净起点）。

**commit** 见 §8 commit 表最后一项。

### M9 · 术语纠错 MCV → MVC

**问题** 术语本应是 **MVC**（Maximum Voluntary Contraction，最大自主收缩，sEMG 归一化金标准），历史代码里混入了 39 处 MCV 错拼。最严重一处：弯举切换后 TTS 对用户播报的「准备好后请说 开始 MCV 测试」—— 用户听到后会跟着念错。

**做法**：
- **TTS 播报文案**改为「开始 MVC 测试」：[voice_daemon.py:1917](hardware_engine/voice_daemon.py#L1917)
- **ASR 关键词保留双拼**（`WAKE_WORDS` 外的命令词匹配列表里 MCV 和 MVC 并存）
  - 即使用户跟着原文案念错，或百度 ASR 把其中一种识成另一种，都能命中
  - 正好给我们一个容错网
- **文档**：权威指南、验收视频排练、对外报告里用户可见文案全部改 MVC
- **内部变量名 `_mcv_wait_until` / `_mcv_peak` / 日志标签 `[MCV窗口]` 暂不改**（改动面大，用户不可见，列入 §13 backlog）

**底层 IPC 文件名本就正确**：`/dev/shm/mvc_calibrate.request` / `.result` / `_MVC_REQUEST_PATH` / `api_mvc_calibrate()` 均用 MVC。

| 警报持续 > 1s 才被掐断 | M1 未命中（拼音 patterns 不够）或 mic 被 aplay 霸占 |

## 12. 硬件通道实测决策表（历史保留）

> 2026-04-18 完成。决策：`plughw:0,0` 作为默认（板载 RK809）。ES7243 I2C 已放弃。

### 四档实测对比

| 设备 | 说明 | 2026-04-18 实测 peak |
|-----|------|----------------------|
| `hw:0,0` | 板载 RK809（含 ES7243 I2C） | < 5（全零，I2C -6 ENXIO）|
| `hw:3,0` | ES7243 直采 | < 5（同样全零）|
| `hw:2,0` | USB 摄像头麦直通 | 不稳定（多机型兼容差）|
| `plughw:2,0` | USB 摄像头麦 + ALSA 转换层 | 3000+（备选降级方案）|
| **`plughw:0,0`** | **板载 RK809 + ALSA 转换层** | **3000+（当前主通道）** |

### peak 阈值说明
- < 5：全零（硬件/驱动不通）
- 5 ~ 1000：底噪级别
- 3000+：正常人声
- 20000+：大声说话 / 饱和
