## 1. 30 秒速读

**系统定位**：边缘板端 AI 健身教练的深蹲评判子系统。输入 = (摄像头骨架角度 + 双通道 EMG)，输出 = (three-class classification + similarity)，驱动 UI 计数 / 骨架染色 / 语音播报。

**三大状态**：
- **pure_vision**：仅视觉 FSM 角度硬判定（`bottom_angle < 100°` → good, 否则 failed）。GRU 不推理。
- **vision_sensor**：FSM 只负责 rep 边界检测 + 疲劳累加。**good / failed / comp 完全由 GRU 三分类决定**，FSM 的"不标准"语音警报被禁用。
- 通过语音"纯视觉" / "视觉加传感" 切换，写 `/dev/shm/inference_mode.json`。

**模型**：`CompensationGRU` (7D × 30 帧窗口，hidden=16，GRU+cls+sim+phase 四头，**10.8 KB**)。MIA Squat 数据集训练，三分类：standard (0) / compensating (1) / non_standard (2)。

**训练产出**：`hardware_engine/extreme_fusion_gru.pt`。

**当前状态**：训练集 acc 98.3%，混淆矩阵清洁，三类自检全绿（用 MIA 真实 CSV 验证）。

## 5. MIA 数据集（单一训练源）

### 5.1 引用

**Chiquier & Vondrick, "Muscles in Action", ICCV 2023**
- Repo: https://github.com/cvlab-columbia/musclesinaction
- 数据：8 通道 EMG + 3D 关节坐标 + 视频
- 52.9GB tar 下载于 `data/mia_squat_raw/MIADatasetOfficial/`

### 5.2 EMG 通道映射（IronBuddy 只用 2 通道）

MIA 原始 8 通道顺序：
```
['rightquad', 'leftquad', 'rightham', 'leftham',
 'rightglutt', 'leftglutt', 'leftbicep', 'rightbicep']
```

IronBuddy 2 贴片硬件 ↔ MIA 索引映射：
- **CH0 (Target) = rightquad** (idx=0) — 深蹲发力主肌（股四头肌）
- **CH1 (Comp) = rightglutt** (idx=4) — 深蹲代偿监测肌（臀大肌）

对应 `mia_preprocess_squat.py:69-74`。

### 5.3 关节角度

MIA `joints3d.npy` 是 SMPL-49 格式：
- 右髋 = idx 1
- 右膝 = idx 4
- 右踝 = idx 7
- 膝角 = angle(hip, knee, ankle) 三点法

### 5.4 预处理产出格式（**权威 10 列 7D CSV**）

```
Timestamp, Ang_Vel, Angle, Ang_Accel, Target_RMS, Comp_RMS, Symmetry_Score, Phase_Progress, pose_score, label
```

- **Ang_Vel**：`np.gradient(knee_angle) * fps` (deg/s)
- **Ang_Accel**：`np.gradient(ang_vel) * fps` (deg/s²)
- **Target_RMS**：`rolling_rms(|rightquad|, 100ms)` — MIA 原始范围 **1.6 – 349**，median ≈ 34
- **Comp_RMS**：`rolling_rms(|rightglutt|, 100ms)` — MIA 原始范围 **1.6 – 373**，median ≈ 38
- **Symmetry_Score**：`min(|R|,|L|) / (max(|R|,|L|)+eps)` — 右 vs 左股四头肌的对称性
- **Phase_Progress**：线性 [0, 1] 覆盖 rep 时长

### 5.5 label 规则（MIA 只生成 2 类）

见 `mia_preprocess_squat.py:infer_label_from_angle`：
- `60° ≤ min_knee_angle ≤ 75°` → **golden** (standard)
- `min_angle < 60°` 或 `> 75°` → **bad** (non_standard，过深 或 半蹲)

### 5.6 当前数据量

```
data/mia/squat/golden/   761 个 per-rep CSV (standard)
data/mia/squat/bad/      203 个 per-rep CSV (non_standard)
                         + compensating 0 → 训练时合成
```

### 5.7 Compensating 合成规则（**必须读，修模型前看**）

在 `train_gru_three_class.py:_synthesize_compensating()`。

**关键设计**：三类特征空间分界必须清晰。
- **Standard**：Target_RMS 高 (30–80 pct)，Comp_RMS 低 (10–30 pct)
- **Compensating**：Target_RMS 低 (15–40 pct)，Comp_RMS 高 (65–95 pct) ← **合成产物**
- **Non_standard**：Target_RMS 低 (10–35)，Comp_RMS 低 (5–25)

**为什么不是简单 `target×0.5 comp×2.0`**：MIA 原 bad 类的 target_rms 均值（55）甚至比 golden（45）更高，两类在 EMG 分布上几乎重叠。如果仅按乘法变换，合成的 compensating 与 golden 在 comp 上严重重叠（都饱和到 1.0），模型学不会区分。所以直接**用线性组合+随机偏置重塑到目标值域**：

```python
# Target 强制压到 15-40 pct
comp[:, 3] = clip(uniform(15, 35) + 10*phase + N(0,3), 10, 45)
# Comp 强制抬到 65-95 pct + 起身相位再加 10
comp[:, 4] = clip(uniform(60, 85) + 15*phase + N(0,4), 55, 100)
phase>0.55 再 + uniform(10) clip to 100
# Symmetry 偏低 (代偿时左右不对称)
comp[:, 5] *= uniform(0.5, 0.75)
```

**从每个 golden rep 派生 1 条 compensating**（`N_COMP_PER_GOLDEN=1`），最终三类数据量：
```
standard:     761 reps  (53% of train windows)
compensating: 761 reps  (41%)
non_standard: 203 reps  (12%)
```

类别权重平衡见 `train_gru_three_class.py` 的 `weights = total / (3 * count)` 逻辑。

### 5.8 MIA 数据的局限（**不绕不过的 Ground Truth**）

1. **每 clip 只有 30 帧**（约 1 秒），是 rep 的单段（下蹲-底部段），**没有完整 squat cycle**。推理时用户做完整 cycle（下蹲 → 起身），angle 分布与训练集不完全一致。
2. **standard vs non_standard 靠 knee angle 区分**，两类 EMG 重叠严重。
3. **compensating 是合成数据**，物理真实性依赖合成公式；真实硬件 EMG 需后续物理校准（见 [§14.4 下午任务](#14-下午任务)）。

### 6.3 训练细节

- **模型**：`CompensationGRU(input_size=7, hidden_size=16, num_layers=1)` — 复用 `fusion_model.py` 定义（不修改）
- **损失**：`0.3 × MSE(similarity) + 0.7 × CrossEntropy(3-class)`
- **Similarity 目标**（训练时与 classification 一起回归）：
  - standard → 1.0
  - compensating → 0.6
  - non_standard → 0.3
- **类别权重**：反比于样本数（`weights[i] = total / (3 * cnt[i])`），防 non_standard 少数类被忽略
- **Scheduler**：`StepLR(step=8, gamma=0.5)`
- **Early stopping**：无（数据量够，20 epoch 足够）
- **耗时**：CPU 约 100 秒（WSL i7）

### 6.4 归一化（**训练/推理必须完全一致**）

```python
def _normalize_inplace(features):
    features[:, 0] = clip(features[:, 0] / 30.0, -3.0, 3.0)   # Ang_Vel
    features[:, 1] = features[:, 1] / 180.0                    # Angle
    features[:, 2] = clip(features[:, 2] / 10.0, -1.0, 1.0)    # Ang_Accel
    features[:, 3] = clip(features[:, 3], 0.0, 100.0) / 100.0  # Target_RMS ← clip 是关键
    features[:, 4] = clip(features[:, 4], 0.0, 100.0) / 100.0  # Comp_RMS   ← clip 是关键
    # Symmetry / Phase 已是 0-1, 不处理
```

**为什么 Target_RMS/Comp_RMS 要先 clip 再除 100**：
MIA 原始 RMS 范围 1.6–349。如果只除 100 不 clip，训练分布是 0.016–3.49，median 0.34。推理时 `muscle_activation.pct / 100` 严格 0–1。两者分布不一致，训练见过的高值 (>1) 在推理永远看不到，模型落到训练分布外侧发懵（历史故障 V7.14 前的 GRU 饱和根因之一）。

### 6.5 Self-test 必须通过

训练末尾会跑 3 个 case，**用 MIA 真实 CSV 抽样构造**（避免手造分布偏移）：

```python
case standard     → 从 data/mia/squat/golden/ 抽一条 CSV, 原样
case compensating → 从 golden 抽一条, 应用 _synthesize_compensating 变换
case non_standard → 从 data/mia/squat/bad/ 抽一条, 原样
```

三个 case 必须 **argmax 正确**，否则**不保存 .pt**（硬闸门）。

## 7. 推理管线 & 归一化契约

### 7.1 模型架构（`fusion_model.py:24 CompensationGRU`）

```
Input: (batch, seq_len=30, 7)
  ↓  (input_size=7 时跳过 proj)
GRU(7 → 16, batch_first=True)
  ↓  取最后时步 hidden: (batch, 16)
  ├─ cos_sim(h, golden_embed) + h → sim_head(17→8→1→Sigmoid) → similarity
  ├─ h → cls_head(16→3)                                       → cls_logits (argmax = classification)
  └─ h → phase_head(16→4)                                     → phase_logits
```

**参数量**：~2700 个 float32 ≈ 10.8 KB 权重文件。

**输出**：
```python
{
    "similarity":     float 0-1,   # 与 "golden embedding" 的相似度
    "classification": str,          # "standard" | "compensating" | "non_standard"
    "confidence":     float 0-1,   # softmax 最大值
    "phase":          str,          # "standing" | "descending" | "bottom" | "ascending"
}
```

### 7.2 推理入口（`main_claw_loop.py:834-916`）

GRU 推理**仅在以下条件满足时触发**：
1. `inference_mode == "vision_sensor"`
2. `_cur_reps > fsm._prev_total_reps_gru`（rep 边界推进，**用 `_total_reps_count`，不用 good+failed**）
3. `_GRU_MODEL is not None`（加载成功）
4. `len(_gru_feature_buf) >= 30`（窗口够用）

每触发一次更新 `fsm._prev_total_reps_gru = _cur_reps` 防重复。

### 7.3 推理端归一化（**与训练严格一致**）

```python
window = np.array(_gru_feature_buf[-30:], dtype=np.float32)
window[:, 1] /= 180.0
window[:, 3] /= 100.0            # ← muscle_activation 已是 0-100 pct
window[:, 4] /= 100.0            # ← 同上
window[:, 2] = np.clip(window[:, 2] / 10.0, -1.0, 1.0)
# Ang_Vel (col 0) 推理端 /30 归一化 — V7.15 已对齐 (训练时也 clip/30)
nn_result = _GRU_MODEL.infer(window)
```

### 7.4 分类后分发（V7.15 新加）

```python
if nn_result:
    cls = nn_result["classification"]
    if cls == "standard":
        fsm.good_squats += 1
    elif cls == "non_standard":
        fsm.failed_squats += 1

    elif cls == "compensating":
        fsm.trigger_buzzer_alert(kind="代偿")

            fsm._compensation_count += 1
            fsm._compensation_last_rep = _cur_reps
```

## 8. FSM 解耦（pure_vision vs vision_sensor）

### 8.1 双路计数器设计

`SquatStateMachine.__init__()` 持有：

| 字段 | 含义 | 推进时机 |
|-----|------|---------|
| `good_squats` | standard rep 数 | **pure_vision**: bottom<100°; **vision_sensor**: GRU=standard |
| `failed_squats` | non_standard rep 数 | **pure_vision**: bottom>100°; **vision_sensor**: GRU=non_standard |
| `_compensation_count` | compensating rep 数 | 仅 vision_sensor: GRU=compensating（带 `_compensation_last_rep` 防重复）|
| `_total_reps_count` | **总 rep 数**（边界计数器，与模式无关）| DESCENDING → STAND 过渡时 += 1 |
| `_prev_total_reps` | DB 日志路径的 rep 进度 | 写 DB 后同步 |
| `_prev_total_reps_gru` | GRU 推理路径的 rep 进度 | 推理触发后同步 |
| `total_fatigue_volume` | 疲劳积分 | **所有** rep 完成时 +1500/7 ≈ 214.3 |

### 8.2 三计数器独立的必然性（**V7.14 历史血泪**）

早期代码把 `_prev_total_reps` **同时**给 DB 路径和 GRU 路径用。DB 路径先 `_prev_total_reps = _cur_reps`，导致 GRU 路径的 `_cur_reps > _prev_total_reps` **永远 False** → GRU 从不触发 → classification 永远缺失 → UI 显示"无模型"。

**V7.14 根因修复**：拆成 `_prev_total_reps`（DB 用）+ `_prev_total_reps_gru`（GRU 用）。

### 8.3 模式切换运行时

```python
# V7.15 在 FSM.update() DESCENDING→STAND rep 结算里读 inference_mode
now_for_mode = time.time()
if now_for_mode - self._mode_last_ts > 0.1:      # 100ms 缓存避免每帧读盘

        with open(...) as f:

if self._mode_cache == "vision_sensor":
    logging.info(f"⏳ [vision_sensor] rep #{self._total_reps_count} 等待 GRU 分类")
    # 不动 good/failed, 不 trigger alert
else:
    # pure_vision: 保留角度硬判定
    if bottom < ANGLE_STANDARD: self.good_squats += 1
    else:                       self.failed_squats += 1; trigger_buzzer_alert()
```

### 8.4 底部角度外插补偿（V7.13）

DESCENDING 分支在每帧比 min 角度之前，先估算"漏帧时可能错过的更深底部"：

```python
if self._last_valid_angle_sq is not None:
    dt = now - self._last_valid_ts
    # 两帧间隙 80-250ms 且上帧在下落
    if 0.08 < dt < 0.25 and self._last_ang_vel_sq < -8.0:

        virtual_bottom = max(40.0, predicted)      # 物理钳位
        self._min_angle_in_rep = min(min_so_far, virtual_bottom, angle)
```

防止 15fps 摄像头在快速下蹲时错过真实底部帧。配合 V7.13 视觉帧率拉到 25fps，rep 误判率从 30% 降到 <10%。

### 10.4 EMG 心跳图 streaming 模式（V7.16）

**行为**：曲线**永不清空**，从右侧不断流入新数据，最老数据从左侧滚出，像 htop CPU 曲线。

**关键要点**：
1. `emgHistory[k]` 初始化就用 0 预填满 `EMG_MAX_POINTS`（200）个元素
2. 每 100ms push 一个新值 + shift 掉最老值，**数组长度恒 = MAX_POINTS**
3. 绘图 `x = (i / (MAX_POINTS - 1)) * W` → 永远铺满 canvas 宽度
4. Canvas 尺寸 `1600 × 220`（逻辑像素），CSS `width:100%` 拉伸铺满面板
5. **会话清零（fatigue reset / 手动 reset）也用 0 预填满**，不退回空数组

**与 V7.15 scope 模式对比**：
- V7.15 scope：进入 DESCENDING 清空 `emgHistory[k] = []`，数组从 0 个点开始累积，铺满 100 个点前曲线只占 canvas 左 `len/100` 宽度（例如 20 个点时只占左 1/5）— 用户反馈"左 1/5 就刷新回空白"就是这个 bug
- V7.16 streaming：永不清空，曲线总是铺满，看单次动作细节靠时间窗口（20s 内能看到 10+ 次 rep 的形状对比）

**实现**：
```javascript
var EMG_MAX_POINTS = 200;
var emgHistory = {
    quadriceps: new Array(EMG_MAX_POINTS).fill(0),
    glutes:     new Array(EMG_MAX_POINTS).fill(0),
    calves:     new Array(EMG_MAX_POINTS).fill(0),
    biceps:     new Array(EMG_MAX_POINTS).fill(0),
};
// push + shift 保持恒定长度
emgHistory[k].push(val);
while (emgHistory[k].length > EMG_MAX_POINTS) emgHistory[k].shift();
// window._emgMaybeResetForNewRep 保留但为 no-op (API 稳定性)
```

## 11. 历史三轮根因突破（V7.13 → V7.15）

### 11.1 V7.13（视觉帧率 + 底部外插）

**问题**：15 fps 摄像头下快速深蹲漏掉真实底部帧，半蹲误判率 30%。

**修复**：
- `cloud_rtmpose_client.py TARGET_FPS` 默认 15 → 25
- `local_yolo_pose.py _INFER_CACHE_TTL` 0.033 → 0.025
- `main_claw_loop.py` FSM 轮询 0.05 → 0.03s（33Hz）
- SquatStateMachine DESCENDING 分支加角度外插补偿

### 11.2 V7.14（三大堵塞根因）

**根因 A — 疲劳上限 UI 显示失灵**：`index.html:2586` 读一个 HTML 里根本不存在的 `cfgFatigueLimit` input，fallback 硬编码 1500。语音改了也不显示。  
**修复**：直接读 `d.fatigue_limit` server 字段。

**根因 B — GRU 永远不触发**：`main_claw_loop.py:819` DB 路径抢先把 `_prev_total_reps = _cur_reps`，line 822 GRU 触发条件 `_cur_reps > _prev_total_reps` 永远 False。  
**修复**：拆 `_prev_total_reps`（DB）和 `_prev_total_reps_gru`（GRU）两个独立计数器。

**根因 C — EMG 饱和 100/100/100**：`domain_calibration.json` 的 α=2.12 β=-21.4 把 simulator 输出的 MIA 域 pct 正变换时放大到 clip 100%。  
**修复**：simulator 加载 `domain_calibration.json` 做反变换（MIA 域 → ESP32 域），udp_emg_server 正变换后回到 MIA 域。

**附加**：`simulator --rate-hz` 1000 → 500（防 SSH 卡死）；`simulator --mvc-assist` 默认 off；新增独立 `simulate_mvc_burst.py`。

### 11.3 V7.15（GRU 饱和 + 解耦 + EMG scope）

**根因 ① GRU 只训练了 2 类**：`train_fusion_network()` 期望 3 个 CSV (`golden/lazy/bad`)，但 MIA 只生成 `golden + bad`，第 3 类（lazy→compensating）缺训练样本，模型对该类权重饱和不学习。  
**修复**：新脚本 `tools/train_gru_three_class.py` 从 golden 派生合成 compensating，三类均衡训练。train acc 98.3%，per-class 96/100/97。

**根因 ② 训练/推理归一化分布不匹配**：训练时 Target_RMS ÷ 100 范围 0.016–3.49；推理时 muscle_activation.pct ÷ 100 范围 0-1。训练见过 >1 推理永远 <1，模型发懵。  
**修复**：训练时 `clip(target_rms, 0, 100) / 100` 统一到 0-1 域。

**根因 ③ vision_sensor 模式 FSM 角度硬判定未禁用**：`main_claw_loop.py:290` trigger_buzzer_alert 对 inference_mode 无感知，两模式都触发"不标准"警报，good/failed 仍被角度硬判定累加。  
**修复**：FSM DESCENDING 分支 100ms 缓存读 mode，vision_sensor 下只推 `_total_reps_count` + 疲劳，不 touch good/failed/comp，不 trigger alert。GRU 推理后按 classification 分发。

**根因 ④ 代偿骨架不染色**：`updateRig` 只看 good/failed 增长触发 flash，vision_sensor 下 compensating 这两者都不涨，骨架无反应。  
**修复**：`updateRig` 加 `comp` 参数；`_rigLastComp` 独立跟踪；comp 增长触发 flash='good' + `_lastClassification='compensating'` 组合 → 黄色 muscleGrad。

**根因 ⑤ EMG 图刷新慢 + 滚动历史混乱**：`index.html` EMG polling 3000ms（3 秒！），且 rolling window 永不清空，看不出单次动作特征。  
**修复**：polling 100ms（10Hz）；state 进入 DESCENDING 清空 `emgHistory`；EMG_MAX_POINTS 60→100（10 秒窗）。

**根因 ⑥ udp_emg_server io_dumper 33Hz 写**：进一步刷新率瓶颈。  
**修复**：`time.sleep(0.03)` → `0.02`（50Hz）。

### 11.4 V7.16（EMG 心跳图 streaming）

**根因 — 曲线只占左 1/5 画布**：V7.15 的 scope 模式进入 DESCENDING 清空 `emgHistory[k]`，数组从 0 个元素开始填。绘图公式 `x = (i / (MAX_POINTS-1)) * W` 在 `arr.length=20` 时最右点 `x = 19/99 × W ≈ 19% W`，所以曲线只画在左侧 1/5。用户直观看到"左 1/5 又刷新回空白"。

**修复**：
1. 取消清空逻辑（`_emgMaybeResetForNewRep` 改成 no-op，保留 API 兼容）
2. `emgHistory` 初始化用 0 预填满 `EMG_MAX_POINTS`（200 个点），一开始就铺满宽度
3. 每次 push 同步 shift，保持数组长度恒 = MAX_POINTS
4. 会话 reset 路径也用 0 预填（不退回空数组）
5. Canvas 尺寸 600×120 → 1600×220（提高清晰度 + 高度变大）
6. 窗口 100 pts → 200 pts（10Hz × 20s，单个画布内看到 10+ 次 rep 的形状对比）

**语义转变**：从"单次动作 scope"改为"连续 streaming 心跳图"。单次动作细节靠窗口滑动时间对比（而非清空），这样用户视觉上看到连续变化而不是中断。

## 13. 常见 Debug 场景

### 13.1 UI 一直显示"无模型"

**症状**：state_feed 无 `classification` 字段。  
**诊断**：
```bash
tail -20 /tmp/fsm_loop.log | grep -E "GRU|prev_total"
```
**可能原因**：
- GRU 模型加载失败（`[GRU] Loaded` 没出现）→ 权重文件缺失或损坏，重 rsync
- 用户没做 rep（GRU 只在 rep 完成时推理，没完成 rep 就没 classification）
- `_prev_total_reps_gru` bug（V7.14 修过，若复发查 main_claw_loop.py 计数器拆分）

### 13.2 GRU 一直输出同一类

**症状**：所有 rep 分类都是 standard（或都是其他某一类），相似度固定。  
**诊断**：
```bash
# 离线重跑 self-test
python3 tools/train_gru_three_class.py --selftest-only
```
**若 self-test 失败**：模型权重坏了，重训（见 §6.1）。  
**若 self-test 成功但板端表现不对**：
- 推理端归一化与训练不一致（检查 `main_claw_loop.py` window 除法与 `train_gru_three_class._normalize_inplace`）
- 模拟器发的数据被 domain_calibration 正变换后分布偏离训练集（查 `/dev/shm/muscle_activation.json` 实际 pct 值）

### 13.3 代偿骨架不染色

**症状**：语音播报"代偿"，顶部"代偿"cell 有数，但中间骨架没变黄色。  
**诊断**：
- `state_feed` 的 `comp` 是否递增（`curl /state_feed | grep comp`）
- `updateRig` 第 7 个参数 `comp` 是否传进去（历史 V7.15 前没传）
- `_lastClassification` 是否在 compensating 时被更新（`index.html:2867`）

### 13.4 EMG 曲线像直线

**症状**：EMG 图全是平线，无起伏。  
**诊断**：
```bash
cat /dev/shm/muscle_activation.json      # 看 pct 值
stat -c "%y" /dev/shm/emg_heartbeat       # 看心跳 mtime
pgrep -af udp_emg_server                  # 进程在不在
```
**可能原因**：
- `udp_emg_server` 被 Killed（系统 OOM 或进程被人 kill）→ 重启 emg 服务
- 模拟器没跑（前端 scope 模式下 state=STAND 时不清空，保持上次曲线）
- 模拟器模式闸门关着（pure_vision 模式下发零）→ 语音切 "视觉加传感"

### 13.5 UI EMG 曲线大幅向上平移

**症状**：肌电条一直顶到 80-100%，完全饱和。  
**诊断**：查 `domain_calibration.json` 的 α/β 是否和 simulator 里反变换用的版本一致。  
**修复**：模拟器加载时会打印 `[DOMAIN_INV] 已加载 ...`，对不上就重 rsync domain_calibration.json 到板端。

### 13.6 服务反复被 Killed

**症状**：log 末尾 `Killed`，pgrep 看不到进程。  
**诊断**：
```bash
ssh toybrick@... 'free -h | head -2; dmesg | tail -20 | grep -i kill'
```
**可能原因**：
- 板端内存紧张（1.9GB 总，多服务挤）
- simulator 1000Hz 压垮（V7.14 降到 500Hz 已修）
- NPU 初始化 peak

### 14.3 推荐弯举训练脚本结构（仿本深蹲的 train_gru_three_class.py）

```
tools/train_gru_three_class_bicep.py
├─ load: data/bicep_curl/{golden, bad, lazy}/*.csv  (已有 10 列 7D 格式)
├─ label map:
│    golden → standard (0)
│    lazy   → non_standard (2)
│    bad    → compensating (1)   ← 弯举的 "bad" 语义是摆臂借力, 正好是代偿

├─ 训练: CompensationGRU, 20 epoch, 同损失函数

```
